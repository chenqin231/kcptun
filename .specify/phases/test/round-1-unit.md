# speckit.test — Round 1：单元测试

> 所属命令：speckit.test | 阶段：Round 1 of 3

## 目标

执行单元测试，定位失败用例到具体 task，**并以 tester 人格主动补充探索性用例**（不仅仅跑已有测试）。

## 执行步骤

### 1a. 单元测试入口

从 Setup 的测试矩阵中读取 Round 1 入口。若标记 `⏭ 跳过`，直接跳到 1d。

### 1b. 执行测试

执行检测到的命令，**不得静默吞错**。完整显示测试输出（通过数、失败数、跳过数、耗时）。

若出现失败用例，对每个失败用例输出：
```
❌ <test name>
  所属 task：T<NN>（从 tasks.md 推断，依据：测试文件路径、测试名关键词）
  失败类型：assertion / error / timeout
  根因猜测：<简述>
  修复建议：<指向哪个文件/函数>
```

### 1c. 结果评估

- **全绿**：记录到本轮 Pass 计数，继续 1d
- **有红**：记录到本轮 Fail 计数；失败用例不阻断后续 round 执行（Round 2/3 仍要跑），但最终 report 会汇总为 NO-GO

### 1d. 探索性用例补充（tester 人格强制输出）

**无论单元测试通过与否，也无论是否跳过 1a-1c，本步都必须执行。**

以破坏性思维为 implement 出的核心模块**至少补充 3 条**未被覆盖的测试场景。格式：

```
🧪 探索性建议 1：空输入 / null / undefined
  目标函数：<module.fn>
  输入：<具体构造>
  预期：<正确行为>
  当前覆盖：❓ 未验证 / ❌ 未覆盖

🧪 探索性建议 2：边界值
  目标函数：<module.fn>
  输入：<具体构造，如 MAX_INT、空数组、单元素>
  预期：<正确行为>
  当前覆盖：...

🧪 探索性建议 3：并发 / 竞态 / 重入
  场景：<具体描述>
  预期：<正确行为>
  当前覆盖：...
```

**额外维度**（可选，鼓励覆盖）：
- 异常路径（网络中断、I/O 失败、权限不足）
- 字符编码 / 特殊字符（emoji、中日韩、RTL、控制字符）
- 时区 / 时钟跳变 / 闰秒
- 幂等性 / 重复调用

### 1e. 本轮汇总

```
📊 Round 1 — 单元测试
  执行：✅ / ⏭
  入口：<命令>
  通过：N
  失败：N
  跳过：N
  探索性建议：N 条（详见上方）
```

**Round 1 完成后**：Read `.specify/phases/test/round-2-coverage.md` 执行 Round 2。
