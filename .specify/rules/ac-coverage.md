# AC 覆盖规则（AC Coverage Rules）

> 每条 FR 的 AC 必须覆盖以下三个维度。

## 维度 1：正常操作路径（Happy Path）

≥1 条 AC（所有 FR 强制）：
- 描述用户按预期流程操作的**完整结果**
- 必须包含：操作 → 结果 → 用户可感知的反馈
- 示例：`用户点击"提交订单" → 订单创建成功 → 页面跳转至订单详情页，顶部显示 Toast "订单提交成功"，持续 3 秒自动消失`

## 维度 2：异常操作路径（Error Path）

≥1 条 AC（所有 FR 强制，视复杂度通常 2-3 条），从以下类别选取：
- **输入异常**：空值 / 格式错误 / 超长 / 非法字符 / 重复值
- **状态异常**：无权限 / 未登录 / 数据不存在 / 数据已过期 / 并发冲突
- **外部异常**：网络中断 / 服务超时 / 第三方接口失败
- 每条异常 AC 必须包含：异常条件 → 系统行为 → 用户可感知的反馈 → 恢复方式
- 示例：`网络中断时点击"提交订单" → 订单不创建 → 页面显示内联错误提示 "网络连接失败，请检查网络后重试"，提交按钮恢复可点击状态`

## 维度 3：消息反馈（Message / Dialog）

**不是独立的 AC 类别**，而是维度 1 和维度 2 中每条 AC 的必填属性。每条 Happy/Error AC 的结果描述中必须明确：
- **反馈形式**：Toast / Modal 弹窗 / Inline 内联提示 / Notification 通知 / 页面跳转 / 状态变更
- **具体文案**：用户看到的确切文字内容（禁止写"提示错误信息"、"显示成功提示"等模糊描述）
- **触发时机**：何时出现（操作后立即 / 异步回调后 / 延迟 N 秒）
- **消失方式**：自动消失（N 秒后）/ 用户手动关闭 / 页面跳转后消失 / 持续显示直到状态变更
- **适用范围**：有用户交互界面的 FR 强制执行；纯后端/批处理 FR 的反馈可简化为日志输出或返回值描述

## AC 完整性检查表（每个 FR 写完后必须逐项确认）

| 检查项 | 要求 | 不达标处理 |
|--------|------|-----------|
| 正常路径 AC | ≥1 条，含操作→结果→反馈 | 补写 |
| 异常路径 AC | ≥1 条，含异常条件→系统行为→反馈→恢复方式 | 补写 |
| 反馈形式 | 每条 AC 的反馈指定了具体形式（Toast/Modal/Inline等） | 补充形式 |
| 反馈文案 | 每条 AC 写明用户看到的确切文字 | 补充文案 |
| 无歧义验证 | 两人独立阅读同一条 AC，能得出相同的 pass/fail 判定 | 重写 |

## AC Coverage 改写参考

| ❌ 不合格（缺覆盖） | ✅ 合格（三维覆盖） |
|---------------------|-------------------|
| 用户可以提交表单 | 正常：用户填写完所有必填项后点击"提交"→ 表单提交成功 → Toast "提交成功" 显示 3 秒后消失，页面跳转至列表页 |
| （无异常 AC） | 异常：必填项为空时点击"提交"→ 表单不提交 → 空字段下方显示红色 Inline 提示 "此项为必填"，焦点定位到第一个空字段 |
| 显示成功提示 | 正常：操作成功后页面顶部显示绿色 Toast "保存成功"，3 秒后自动消失 |
| 提示错误信息 | 异常：服务端返回 500 时显示红色 Toast "系统繁忙，请稍后重试"，5 秒后自动消失，用户可重新操作 |
