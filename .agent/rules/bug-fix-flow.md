# Bug 修复流程与 TDD

> **触发信号**：Bug 报告、测试驱动、修复流程、根因定位
> **对应铁律**：4️⃣ 测试驱动模式 + 5️⃣ 规范自进化

---

## 问题修改流程（TDD 强制）

```
发现问题 → 【写失败测试】 → 复现问题 → 定位根因 → 验证根因 → 修复代码 → 【验证测试通过】
```

**关键约束**：
1. ✅ **必须先写测试**：修复前编写能复现 Bug 的测试
2. ✅ **确认测试失败**：运行测试，确认失败（证明测试有效）
3. ✅ **修复后测试通过**：修复后测试必须变为通过
4. ✅ **提交包含测试**：Git 提交同时包含修复代码和测试

**禁止**：先写代码 → 测试通过 → 提交（顺序错误）

---

## Bug 修复 TDD 流程

```
步骤1：【写失败测试】 → 创建测试文件，编写复现场景
步骤2：【运行测试】   → 确认测试失败（红色 ❌）
步骤3：【修复代码】   → 编写最小修复代码
步骤4：【运行测试】   → 确认测试通过（绿色 ✅）
步骤5：【重构优化】   → 可选，保持测试通过
步骤6：【提交代码】   → 包含测试文件
```

## 新功能 TDD 流程

```
步骤1：【写测试框架】 → 定义预期行为
步骤2：【运行测试】   → 确认失败（功能未实现）
步骤3：【实现最小功能】→ 让测试通过
步骤4：【补充边界测试】→ 异常/边界场景
步骤5：【重构优化】   → 保持测试绿色
```

## 根因定位（强制）

### 第一步：复现问题
- 记录复现步骤
- 记录错误信息/日志
- 记录预期 vs 实际行为
- 确保能稳定复现

### 第二步：定位根因
- 添加日志追踪执行路径
- 编写测试验证假设
- 使用调试工具单步调试
- 分析错误堆栈
- 使用 `git bisect` 定位引入问题的提交

### 第三步：验证根因
- 测试验证假设是否正确
- 确认修复方案能解决问题
- 评估修复影响范围

### 禁止的做法
- ❌ 未定位根因就修改代码
- ❌ 猜测式修改（"可能是这里，改改试试"）
- ❌ 不验证修复就提交

## CI/测试失败自主修复边界

### 可自主修复（直接修复，修复后告知用户）

| 失败类型 | 修复动作 |
|----------|---------|
| Lint/格式错误 | 运行格式化工具 |
| 明显类型错误 | 添加/修正类型声明 |
| Import 缺失/冗余 | 添加或移除 import |
| 断言值过时 | 更新断言值，附注原因 |
| 依赖版本冲突 | 重新安装依赖 |

### 需要用户审批（停止并汇报）

| 失败类型 | 原因 |
|----------|------|
| 业务逻辑变更 | 可能涉及需求理解偏差 |
| 架构/接口调整 | 影响范围大 |
| 安全相关失败 | 安全问题不可自行决策 |
| 数据库迁移失败 | 不可逆操作 |
| >5 个测试连锁失败 | 需要根因分析 |

**执行原则**：
- 自主修复后必须重新运行完整测试套件
- 修复变更必须在提交信息中说明
- 同一类失败 ≥3 次 → 停止自主修复，报告根因

## 规范自进化

### 触发条件

1. 用户纠正 AI 的错误/遗漏
2. 用户多次强调同一要求
3. 对话结束时检测到规范盲区

### 自检流程

```
用户纠正 AI
  → 【识别根因】违反了哪条规范？规范是否缺失？
  → 【提议更新】确定位置、起草内容、向用户展示
  → 【用户确认】同意 → 执行更新 | 拒绝 → 记录
  → 【回显变更】输出变更摘要
```

### 对话结束时主动检查

- [ ] 本次对话是否暴露规范盲区？
- [ ] 用户是否多次纠正同类问题？
- [ ] 是否有值得记录的经验教训？

任意一项为"是"，输出提议：
```
"检测到规范优化点：
- 问题：xxx
- 建议更新：xxx
- 是否需要更新规范？"
```
