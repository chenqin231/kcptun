# 代码审查规范

> **触发信号**：PR 审查、合并前检查、Code Review

---

## 合并前强制要求

**✅ 提交者必须确保**：
- 功能开发在独立分支（禁止直接在 master 开发）
- 所有自动化测试通过
- 代码通过 Linter 检查，零警告
- 至少一人代码审查
- 所有冲突已解决
- 文档已同步更新
- 合并后删除特性分支

**❌ 绝对禁止**：
- 未经审查直接合并
- 合并包含冲突的代码
- 合并未通过测试的代码
- 合并包含调试代码（`console.log`, `var_dump`, `fmt.Println`）
- 绕过审查流程

---

## 审查者职责

- ✅ 检查代码逻辑正确性
- ✅ 检查编码规范
- ✅ 检查安全隐患
- ✅ 检查性能问题
- ✅ 检查测试覆盖
- ✅ 检查文档同步
- ✅ 提供建设性建议

---

## 审查清单

### 功能正确性
- [ ] 实现了需求中的所有功能点
- [ ] 边界情况正确处理
- [ ] 异常情况正确处理
- [ ] 测试充分且通过

### 代码质量
- [ ] 结构清晰，易于理解
- [ ] 命名规范，语义明确
- [ ] 没有重复代码
- [ ] 没有过度设计

### 安全性
- [ ] 输入验证充分
- [ ] 输出正确转义
- [ ] 没有 SQL 注入风险
- [ ] 没有 XSS 风险
- [ ] 敏感信息没有泄露

### 性能
- [ ] 没有明显性能问题
- [ ] 数据库查询已优化
- [ ] 没有 N+1 查询

### 可维护性
- [ ] 注释充分且准确
- [ ] 文档已更新
- [ ] 配置已说明
