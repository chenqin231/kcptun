# Tasks 拆解与 Develop 执行规范

> **触发信号**：任务拆解、开发阶段、验收报告
> **对应铁律**：2️⃣ 任务拆解阈值 + 3️⃣ 防御性交付

---

## 任务拆解阈值

满足**任意一项**时必须拆解：
- 改动超过 **50行代码**
- 涉及 **多个独立功能模块**
- 预计开发时间 > **2小时**
- 需要 **多步验证** 才能确认正确性

**拆解原则**：
1. 按功能模块切分，每段完成一个独立功能
2. 可独立测试
3. 逻辑边界清晰
4. 依赖关系明确（先基础模块，再依赖模块）

## Tasks 阶段强制模板（5 要素缺一不可）

**[BLOCKING CONSTRAINT]**

```markdown
### Task N: [明确的任务标题 - 祈使句]

**🎯 目标**（一句话说清楚要达成什么）：
- [简洁的目标描述，避免"实现XX功能"等模糊表述]

**📋 具体步骤**（可直接执行，无需额外解读）：
1. [第一步：如"在 src/api/user.ts 中新增 register 方法"]
2. [第二步：如"调用 database.insert() 插入用户记录"]

**📁 涉及文件**（明确修改范围和内容）：
- `path/to/file1.ts` - [新增 XX 函数 / 修改 YY 逻辑]

**✅ 验收标准**（可客观验证）：
- [ ] 功能验证：[如"注册接口返回 201 和用户 ID"]
- [ ] 测试验证：[如"user.test.ts 中的 register() 测试通过"]
- [ ] 代码验证：[如"Linter 无警告"]

**🔗 参考上下文**（强制关联需求和设计）：
- Requirements 章节：[XX]
- Plan 设计决策：[YY]
- 依赖任务：[Task ID]
```

**禁止模糊描述**：
- ❌ "优化 XX"、"完善 YY"、"处理 ZZ"
- ❌ "实现用户注册功能"（太宽泛）
- ✅ "实现用户注册 API 端点（POST /api/register）"

**任务粒度**：单个任务 **1-4 小时**，超过必须继续拆分

## Tasks 阶段审批流程

**[BLOCKING CONSTRAINT]** — AI 输出 tasks.md 后，**禁止直接进入开发**。

**步骤 1：AI 自查**
- 每个任务是否包含 5 要素？
- 是否有模糊描述？
- 任务粒度是否合理？
- 验收标准是否具体可验证？

**步骤 2：输出审批请求**

```markdown
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  📋 Tasks 拆解完成，请审批
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

共 [N] 个任务。请逐条检查：
1. ✅ 5 要素完整？
2. ✅ 步骤具体可执行？
3. ✅ 验收标准明确？
4. ✅ 粒度合适（1-4 小时/任务）？
5. ✅ 有无遗漏或冗余？
```

## 防御性交付清单

每次代码交付必须附带：

### 📋 潜在 Bug 清单（五个维度）

1. **异常处理** — 捕获异常？重试机制？
2. **边界条件** — 空数组？超大文件？特殊字符？
3. **并发问题** — 多用户冲突？竞态条件？
4. **资源泄漏** — 文件句柄？数据库连接？内存？
5. **性能隐患** — N+1 查询？分页？无限循环？

### ✅ 测试用例清单

- **正常场景**（Happy Path）
- **异常场景**（Error Path）
- **边界场景**（Edge Cases）

### 🔍 交付自检

自问：资深工程师会通过这段代码吗？用证据证明可用。

### 🔄 实现回顾（非简单修复时）

自问：是否有更优雅的实现？是则重构，否则记录技术债。

## 完整执行流程

```
[Tasks 阶段]
  → AI 输出 tasks.md → 自查 → 审批请求 → 【等待用户审批】
  ↓ 用户确认通过
[Develop 阶段]
  For 每个任务：
  ├─ 重新阅读上下文
  ├─ 输出理解确认 → 【等待用户确认】
  ├─ 开始写代码
  ├─ 对照验收标准自查
  ├─ 输出验收报告 → 【等待用户验收】
  ├─ 更新 context.md + progress.json
  ├─ 提交代码（git commit）
  └─ 更新 tasks.md 状态
```

**三个关键卡点**：
1. Tasks 输出后 → 等待审批
2. 每个任务执行前 → 等待理解确认
3. 每个任务完成后 → 等待验收确认
