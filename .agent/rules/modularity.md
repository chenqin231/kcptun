# 文件拆分决策树（通用规则）

> 本规则对所有语言生效。语言特定的拆分模式请查看对应的 `*-patterns` skill。

## 行数阈值（强制）

| 等级 | 行数 | 行为 |
|------|------|------|
| 正常 | ≤ 200 | 理想状态 |
| 警告 | 201-300 | 提示建议拆分 |
| 强制 | 301-500 | 必须说明理由或拆分 |
| 硬限 | > 500 | 阻塞，禁止写入 |

豁免：配置文件（.json/.yaml/.toml/.lock）、测试文件（*test*/*spec*）、自动生成代码（*.generated.*）、模板文件（templates/）

## Plan 阶段：文件清单（强制输出）

任何 Plan/设计文档必须包含文件结构规划表：

```markdown
| 文件路径 | 职责（一句话） | 预估行数 | 新建/修改 | 依赖 |
|---------|-------------|---------|----------|------|
```

检查项：
- 每个文件预估 ≤ 300 行
- 每个文件只有一个主要职责
- 无循环依赖

## 写代码前：拆分决策树

```
要实现什么功能？
    │
    ├── 涉及几个关注点？（数据/逻辑/展示/IO/配置）
    │     │
    │     ├── 1 个 → 单文件，预估行数 ≤ 200 → ✅
    │     │           预估行数 > 200 → 按子职责拆分
    │     │
    │     ├── 2-3 个 → 每个关注点一个文件
    │     │
    │     └── > 3 个 → 创建子目录，index 文件统一导出
    │
    └── 修改现有文件？
          │
          ├── 当前文件 ≤ 200 行 → 直接修改
          ├── 当前文件 201-300 行 → 修改前评估是否可提取
          └── 当前文件 > 300 行 → 先拆分再修改
```

## 五条拆分规则

### 规则 1：职责分离（最优先）

一个文件包含 ≥ 2 个不相关的职责 → 拆成独立文件。

判断标准：能否用一句话描述这个文件的职责？不能 → 需要拆。

### 规则 2：行数预警

预估或实际 > 200 行 → 检查是否可提取子模块。

常见可提取对象：
- 工具函数 → `utils.xx` / `helpers.xx`
- 类型定义 → `types.xx`
- 常量/配置 → `constants.xx`
- 校验逻辑 → `validators.xx`

### 规则 3：行数硬限

预估或实际 > 300 行 → 必须拆分。

拆分后每个文件的目标：100-200 行。

### 规则 4：函数膨胀

单函数 > 30 行 → 提取子函数。
单函数 > 50 行 → 考虑提取到独立文件。

### 规则 5：依赖扇入

≥ 3 个文件 import 同一段工具代码 → 提取为独立 utils 文件。

## 拆分维度优先级

按以下顺序决定拆分方式：

```
1. 按职责       → 每个文件只做一件事（最优先）
2. 按抽象层次   → 接口 / 实现 / 类型定义 分离
3. 按变更频率   → 稳定代码 vs 频繁改动代码
4. 按公开/私有  → 对外暴露 vs 内部实现
```

## 常见拆分模式（语言无关）

| 场景 | 拆法 |
|------|------|
| 一个文件又有数据模型又有业务逻辑 | 分离 model + service |
| 一个文件又有路由又有处理逻辑 | 分离 router + handler |
| 工具函数散落在业务代码中 | 提取 utils 文件 |
| 常量/配置混在逻辑里 | 提取 constants / config 文件 |
| 类型定义和实现混在一起 | 提取 types / interfaces 文件 |
| 校验逻辑嵌入业务流程 | 提取 validators 文件 |

## 修改超限文件的流程

当需要修改一个已超过 300 行的文件时：

```
1. 分析文件中有哪些职责
2. 制定拆分方案（目标：每个文件 100-200 行）
3. 先提交拆分（纯重构，不改行为）
4. 再在拆分后的文件上添加新功能
```

**禁止**：在已超限的文件上继续追加代码。
