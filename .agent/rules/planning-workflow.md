# 结构化开发工作流（Planning 模式）

> **触发信号**：规划、设计、新功能、speckit、技术选型
> **对应铁律**：1️⃣ 方案先行原则

---

## 工作流链条（强制按序执行）

```
【创建 specs 目录】
    ↓
Research → 【审批✅】 → Requirements → 【审批✅】 → Plan → 【审批✅】 → Tasks → 【审批✅】 → Develop
  (研究)    (检查点1)      (需求)        (检查点2)    (设计)   (检查点3)    (任务)    (检查点4)    (开发)
```

**审批机制**（强制）：
- 每个阶段完成后，AI 必须暂停并等待用户审批
- 用户审批通过后，才能进入下一阶段
- 如果用户拒绝，AI 需要根据反馈修改当前阶段文档

---

## 适用场景判断

**入口边界由用户决定**。下面的代码行数、文件数仅作参考，AI 在场景模糊时**口头建议**用户选择入口，**不自动跳转**。

### 入口对照表（四档）

| 入口 | 参考触发场景 | 副作用 |
|------|-------------|--------|
| `/speckit.specify`（完整，5 阶段含 Research） | 全新功能 / 引入新第三方库或框架 / 未知技术领域 | 建 specs/ + worktree + 多阶段审批 |
| `/speckit.specify`（精简，4 阶段跳过 Research） | 现有功能优化 / 架构调整 / 复杂 Bug 修复 / 数据库或 API 变更 | 同上 |
| **`/plan`（轻量，inline）** | 单文件改动 / 小型 Bug 修复 / 小型重构 / 不需要新分支 | **零副作用**：inline 输出 + 等用户确认 |
| 直接动手 | 文档/注释更新 / 配置微调 | 无规划 |

### 行数参考（非硬约束）

- `/speckit.*` 完整模式：>200 行 或 >5 文件 或 跨多模块
- `/speckit.*` 精简模式：50-200 行 或 2-5 文件
- `/plan`：<50 行 或 单文件
- 直接动手：<10 行 单文件

### /plan vs /speckit.specify 的核心区别

| 维度 | /plan | /speckit.specify |
|------|-------|------------------|
| 文件副作用 | 零 | 建 specs/、可能建 worktree、写 spec.md |
| 阶段数 | 1（输出方案 + 等确认） | 4-5（Research → Spec → Plan → Tasks → Implement） |
| 审批点 | 1 次（方案确认） | 4-5 次（每阶段一次） |
| 适用代价 | 低 | 高 |
| 适用产出 | 小型修改、Bug 修复 | 完整功能、需归档可追溯 |

## 阶段0：创建 Specs 目录（自动执行）

```bash
# 1. 读取现有编号，自动递增
max_num=$(ls -d specs/[0-9]* 2>/dev/null | sed 's/specs\///' | cut -d'-' -f1 | sort -n | tail -1)
new_num=$(printf "%03d" $((10#$max_num + 1)))

# 2. 创建目录和文档骨架
mkdir -p specs/${new_num}-<需求名>
touch specs/${new_num}-<需求名>/{research,spec,plan,tasks,changelog}.md

# 3. 更新 progress.json
```

## 阶段1：Research（研究阶段）

**产出**：`specs/<编号>-<需求名>/research.md`

**核心内容**：
- 技术选型（候选方案对比）
- 竞品分析
- 可行性分析（技术/时间/风险）
- 依赖分析

**检查点**：
- [ ] 至少对比 2 个技术方案
- [ ] 给出明确推荐方案和理由
- [ ] 评估可行性和风险
- [ ] 列出所有依赖

---

## 阶段2：Requirements（需求阶段）

**产出**：`specs/<编号>-<需求名>/spec.md`

**🔴 需求沟通强化（写文档前必须先确认）**：

```markdown
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  📋 需求确认（写文档前必须先确认）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

我理解的需求要点：
1. ✅ 要做什么：<AI 理解的核心需求>
2. ✅ 目标用户：<谁会使用>
3. ✅ 预期效果：<期望结果>

范围限定：
4. ❌ 不做什么：<明确排除>
5. ❌ 不在本次范围：<延后处理>

❓ 需要确认的问题：
- 问题1：<歧义点>
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**禁止**：未经确认直接编写 spec.md | AI 自行假设需求细节

**核心内容**：业务场景、功能列表、范围限定、验收标准、非功能需求

**检查点**：
- [ ] 业务场景清晰
- [ ] 功能列表完整
- [ ] 验收标准明确
- [ ] 非功能需求已考虑

---

## 阶段3：Plan（设计阶段）

**产出**：`specs/<编号>-<需求名>/plan.md`

**核心内容**：
- 架构设计（系统架构、模块职责）
- API 定义（接口、参数、响应）
- 数据库设计（表结构、字段）
- 界面原型（字符原型或描述）
- 一致性审查（与 Research/Requirements 对照）

**检查点**：
- [ ] 架构设计合理
- [ ] API 定义完整
- [ ] 数据库设计规范
- [ ] 与前两阶段保持一致

---

## 阶段4：Tasks（任务分解）

**产出**：`specs/<编号>-<需求名>/tasks.md`

> 任务模板和拆分标准见 [task-execution.md](task-execution.md)

**检查点**：
- [ ] 任务粒度合适（1-4 小时/任务）
- [ ] 任务独立可测试
- [ ] 优先级明确
- [ ] 依赖关系清晰

---

## 阶段5：Develop（开发阶段）

**开发模式**：增量开发（强制）

```
选择 Task → 编写代码 → 单元测试 → 集成测试
    ├─ ✅ 通过 → 提交 Git → 更新 tasks.md → 下一个 Task
    └─ ❌ 失败 → 修复问题 → 重新测试
```

**核心原则**：
1. 每次只开发一个 Task
2. 完成一个 Task 必须测试
3. 测试通过后才能开启下一个 Task
4. 每个 Task 完成后必须提交 Git

**检查点**：
- [ ] 每个 Task 都有测试
- [ ] 代码符合规范
- [ ] 文档已更新
- [ ] Git 提交规范
