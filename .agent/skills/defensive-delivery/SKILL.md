# defensive-delivery — 代码交付 4 件套

## 何时激活

代码变更已完成，**即将**向用户呈现结果时。无论变更大小（除纯文档改动），都应走本 skill。

## 必须输出的 4 件套

### ① 潜在 Bug 清单（至少 3 条）

按以下维度逐一审查，写出实际发现的风险或「已检查，无发现」：

- **异常处理**：未捕获的异常、吞掉错误的 catch
- **边界条件**：空值 / 0 / 负数 / 空集合 / 超长输入
- **并发**：race condition、资源竞争、死锁可能
- **资源泄漏**：未关闭的文件/连接/订阅
- **性能**：N+1 查询、不必要的循环、大对象内存占用

### ② 测试用例清单（必须可执行）

必须给出具体命令或测试代码骨架，覆盖：

- **正常路径**：1-2 个典型输入 + 预期输出
- **异常路径**：至少 1 个错误场景（非法输入 / 外部失败）
- **边界**：至少 1 个边界值（空 / 最大 / 最小）

例如：
```bash
# 正常
curl -X POST .../api/user -d '{"name":"alice"}' | jq .id   # 期望: 返回 uuid

# 异常
curl -X POST .../api/user -d '{}' | jq .error   # 期望: "name required"

# 边界
curl -X POST .../api/user -d "{\"name\":\"$(printf 'x%.0s' {1..256})\"}" # 期望: 422
```

### ③ 交付自检（用证据证明）

回答："一位资深工程师 review 这段代码时，会通过吗？"

**必须给出证据**，不接受空口：
- ✅ 已按 linter/formatter 走过（附命令输出片段）
- ✅ 已跑相关测试并通过（附测试命令与结果）
- ✅ 关键决策在代码注释或提交信息里有说明
- ⚠️ 或明确列出**可能被拒**的点及原因

### ④ 实现回顾（非简单修复时必答）

回答："有没有更优雅的实现？"

至少考虑一次：
- 能否用现有工具 / 库 / 函数替代自己写的代码？
- 能否拆解为更小、更易测的函数？
- 抽象层次是否合适（不过度 / 不过简）？

纯文档修复或 <10 行的 trivial 改动可省略此项。

## 纯文档改动的精简模式

仅修改 `.md` / 注释 / 配置说明时：
- 省略 ① ② ③
- 仍需 ④（文档结构、措辞是否最优）

## 与其他 skill 的边界

| Skill | 范围 | 粒度 |
|-------|------|------|
| `defensive-delivery` | **每一次**代码呈现 | 单次变更 |
| `/accept` | 阶段/任务级验收 + 上下文清洗 | 多次变更汇总 |
| `fix` skill Phase 6 | Bug 修复流程中的"审查"阶段 | 修复专用 |
| `code-reviewer` agent | 二次独立审查 | 跨视角 |

本 skill 是**作者自检**，不替代 `/accept` 的流程编排，也不替代独立 reviewer。

## 为什么叫"防御性"

"防御"指的是：在把代码递给用户（或合入主干）之前，把所有自己能发现的问题先主动暴露出来。与其等 review 时被挑错，不如先把已知的弱点和测试覆盖呈现出来，让对方评估风险。
