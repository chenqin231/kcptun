# /restore — 丢弃未提交修改（回到上次存档点）

## 用法

- `/restore` — 丢弃所有未提交的修改，回到最近一次 commit 状态

参数：$ARGUMENTS（未使用）

---

## 执行流程

**[BLOCKING CONSTRAINT]** 以下每个步骤都必须实际执行，禁止跳过。

### 步骤 1：检查当前状态

使用 Bash 工具执行 `git status --porcelain`，判断是否有未提交的修改。

如果输出为空（无任何修改），直接输出提示并终止：
```
[/restore] 提示：当前没有未提交的修改，无需还原
```

### 步骤 2：展示即将丢弃的修改

**[MUST - 必须让用户知道将丢弃什么]**

使用 Bash 工具执行以下命令，展示变更摘要：

```bash
git diff --stat
git diff --cached --stat
```

输出格式：
```markdown
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  以下修改将被丢弃
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

<git diff --stat 输出>
<git diff --cached --stat 输出（如有暂存区变更）>

共 <N> 个文件受影响
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 步骤 3：执行还原

使用 Bash 工具执行以下命令：

```bash
# 丢弃工作区修改
git checkout .
# 撤销暂存区（如有）
git reset HEAD .
# 清理未跟踪文件（仅在有未跟踪文件时）
git clean -fd
```

### 步骤 4：验证还原结果

使用 Bash 工具执行 `git status --porcelain`，确认工作区已干净。

如果仍有残留修改，输出警告：
```
[/restore] 警告：还原后仍有残留文件，请手动检查：
<残留文件列表>
```

### 步骤 5：输出还原报告

```markdown
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  已还原到上次存档点
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

当前 HEAD：<commit SHA 前 7 位> - <commit message 首行>
丢弃文件：<N> 个
还原时间：<当前时间>

使用 /load 恢复开发上下文 | /save 保存新的存档点
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 自检清单

**[BLOCKING]** 输出还原报告前，逐项确认：
- [ ] 已展示即将丢弃的修改清单？
- [ ] git checkout . 已执行？
- [ ] git status 确认工作区已干净？

如有未完成项，立即补做后再输出报告。
