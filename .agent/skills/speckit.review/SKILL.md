> 调用 `/speckit.review` 将自动转发到 `/speckit.analyze`。

```text
$ARGUMENTS
```

You **MUST** consider the user input before proceeding (if not empty).

## 定位

**SMA 三维质量检测器**——检查规格文档是否"精确"。

**职责边界**：本 agent 仅负责单文档内的 SMA 质量检测。
跨文档一致性检查（spec↔plan、plan↔tasks 等）由 speckit.analyze 负责。

与 `speckit.analyze` 互补：
- `speckit.analyze` → 检查"是否完整"（覆盖率、术语一致、constitution 对齐）+ 跨文档一致性验证
- `speckit.review` → 检查"是否精确"（每条描述是否只有一种理解方式，单文档内检测）

**核心原则**：
- **S (Specific)**：描述只有一种理解方式，无歧义
- **M (Measurable)**：验收标准可客观判定通过/失败
- **A (Atomic)**：每条任务做且只做一件事，不可再拆

## 适用阶段

自动检测当前文档状态，决定检查范围：

| 检测到的文档 | 检查范围 | 检查维度 |
|-------------|---------|---------|
| spec.md（或 requirements.md）+ plan.md | Plan 阶段 | S + M（需求+设计层面） |
| spec.md + plan.md + tasks.md | Tasks 阶段 | S + M + A（全维度） |
| 仅 tasks.md | 任务审查 | S + M + A（仅任务层面） |
| 仅 spec.md（或 requirements.md） | 需求审查 | S + M（仅需求层面） |

## 执行流程

### 步骤 1：定位文档

按以下优先级查找规格文档：

1. **如果 SpecKit 环境可用**（`.specify/scripts/bash/check-prerequisites.sh` 存在）：
   - 先读取 `specs/progress.json`，提取 `branch` 字段值作为 `<BRANCH_NAME>`
   - 如果 progress.json 存在且 branch 非空：运行 `.specify/scripts/bash/check-prerequisites.sh --json --paths-only --branch <BRANCH_NAME>`
   - 如果 progress.json 不存在：运行 `.specify/scripts/bash/check-prerequisites.sh --json --paths-only`（降级为自动检测）
   - 解析 FEATURE_DIR，从中读取文档

2. **如果 `specs/progress.json` 存在**：
   - 读取 `specsDir` 字段定位文档目录

3. **兜底**：扫描 `specs/` 下最近修改的子目录

在定位到的目录中，查找以下文件（按存在情况决定检查范围）：
- `spec.md` 或 `requirements.md`（兼容两种命名）
- `plan.md`
- `tasks.md`

如果一个文档都找不到，报错终止。

### 步骤 2：加载项目架构上下文（动态发现）

**[不硬编码任何项目特定信息]**

读取以下文件（如果存在）以理解项目架构：

1. 项目根目录的 `CLAUDE.md` 或 `.agent/instructions.md`
2. `.agent/user.md`
3. `.specify/memory/constitution.md`（如果存在）

从中提取：
- **分层结构**：项目使用什么架构模式（MVC / 分层 / Clean / 事件驱动等）
- **命名约惯**：关键类型的命名模式（如 `*Service`、`*Controller`、`*Handler`）
- **通信模式**：模块间如何交互（HTTP / 消息队列 / 函数调用等）
- **错误处理模式**：项目惯用的错误处理方式

如果无法获取架构信息，跳过架构一致性子检查，只执行 SMA 纯文本检测。

### 步骤 3：S（Specific）具体性检测

**核心问题：每条描述是否只有一种理解方式？**

#### S1: 模糊动词检测

扫描文档中的以下动词模式（没有明确终态）：

```
模糊动词清单（可扩展）：
优化|完善|处理|改进|调整|增强|支持|管理|确保|
改善|提升|加强|整合|梳理|重构（无具体目标时）
```

**判定规则**：
- 动词后有明确终态描述 → 通过（如"优化查询，将响应时间从 2s 降到 200ms"）
- 动词后无终态或终态模糊 → 标记为 S1 问题

#### S2: 隐式主语/宾语检测

扫描缺少明确主语或宾语的描述：

```
典型模式：
- "需要重启" → 谁重启？重启什么？
- "返回错误" → 返回给谁？通过什么通道？
- "进行校验" → 谁校验？校验什么输入？
- "通知用户" → 通过什么方式通知？（邮件/推送/页面提示？）
- "保存数据" → 保存到哪里？（内存/文件/数据库？）
```

#### S3: 未定量词检测

扫描缺少具体数值的程度描述：

```
未定量词清单：
性能良好|快速响应|高可用|高并发|安全可靠|
灵活配置|易于扩展|及时处理|大量数据|少量请求|
适当的|合理的|足够的|必要的
```

**判定规则**：
- 后接具体指标 → 通过（如"高可用，SLA 99.9%"）
- 无具体指标 → 标记为 S3 问题

#### S4: 歧义引用检测

扫描指代不明的引用：

```
典型模式：
- "前端显示" → 哪个前端？哪个页面/组件？
- "后端处理" → 哪个服务？哪个接口？
- "配置修改" → 哪个配置文件？哪个字段？
- "该模块" / "这个服务" → 当上下文中存在多个模块/服务时
- "相关文件" → 具体是哪些文件？
```

#### S5: 多解读检测

识别存在 >=2 种合理理解的描述：

- 同一条需求/任务中包含"或"/"也可以"/"可选"但未明确选择
- 条件语句不完整："如果 X 则 Y"但未说明"否则"的行为
- 范围边界不清："用户相关数据"（包括哪些？排除哪些？）

#### S6: 架构一致性检测（依赖步骤 2 的上下文）

仅当步骤 2 成功提取了架构信息时执行：

- 新增组件/文件的位置是否符合项目分层约惯？
- 命名是否与现有模式一致？
- 通信方式是否与现有模式一致？
- 错误处理是否与现有模式一致？

**如果步骤 2 未获取架构信息，跳过此子维度。**

### 步骤 4：M（Measurable）可衡量性检测

**核心问题：验收标准能否客观判定"通过"或"不通过"？**

#### M1: 验收标准缺失

- 需求/任务有功能描述但无验收条件
- 用户故事缺少 Acceptance Scenarios
- 任务缺少"完成"的判定标准

#### M2: 主观判定标准

扫描无法客观判断的验收标准：

```
典型模式：
- "用户体验提升" → 提升多少？怎么衡量？
- "界面美观" → 谁来判断？标准是什么？
- "性能可接受" → 可接受的阈值是多少？
- "代码质量良好" → 哪些指标？什么阈值？
```

#### M3: 缺少预期值

操作的结果缺少具体预期：

- "返回错误" → 什么 HTTP 状态码？什么错误消息体？
- "记录日志" → 什么日志级别？包含哪些字段？
- "发送通知" → 什么内容？什么触发条件？
- "默认启用" → 默认值在哪里定义？用户如何修改？

#### M4: 隐式副作用

操作 A 完成后对其他部分的影响未说明：

- 修改配置后 → 是否需要重启？热加载？
- 删除实体后 → 关联数据怎么处理？（级联删除？软删除？报错？）
- 接口变更后 → 调用方是否需要适配？向后兼容？

#### M5: 不可测试

验收标准在当前条件下无法验证：

- 依赖外部环境但未说明如何搭建测试环境
- 需要特定时间条件（"凌晨3点触发"）但未说明如何加速验证
- 涉及边缘硬件/网络条件但未说明模拟方法

### 步骤 5：A（Atomic）原子性检测（仅 tasks.md）

**核心问题：每条任务是否做且只做一件事？**

#### A1: 复合任务检测

单条任务包含多个独立操作：

```
典型模式：
- "实现 XX 并且 YY" / "实现 XX，同时 YY"
- "创建 XX 以及 YY"
- 单条任务描述超过 3 个动词
- 单条任务涉及 3 个以上文件路径
```

**判定规则**：如果 XX 和 YY 可以独立完成和独立测试 → 应拆为两条任务

#### A2: 跨边界检测

单条任务跨越多个模块/服务/层级：

- 同时修改前端和后端
- 同时修改业务逻辑层和数据访问层
- 同时修改多个微服务
- 同时修改 API 定义和 API 实现

#### A3: 粒度过粗检测

任务描述涵盖的范围过大：

- "实现用户模块" → 应拆为注册、登录、资料管理等独立任务
- "完成数据库设计" → 应按实体拆分
- 单条任务无法在一次提交中完成

#### A4: 依赖内嵌检测

任务隐含了未列出的前置步骤：

- "在 XX 基础上实现 YY" → XX 是否已有独立任务？
- "调用 XX 接口" → XX 接口的创建是否已有独立任务？
- "使用 XX 库" → XX 库的安装/配置是否已有独立任务？

### 步骤 8：自动修复（可选）

报告输出后询问用户：
- "是否需要我自动修复严重问题？"
- 如果用户同意，按报告中的"建议改为"逐条修复并更新文档
- 修复后重新运行检测，确认问题已消除

## 行为规则

- **只读优先**：默认不修改任何文件，仅输出报告。修复需用户明确同意
- **"建议改为"必须具体**：不能只说"请明确"，必须给出一个具体的改写示例
- **不生造问题**：如果文档质量良好，输出"未发现严重问题"并给出统计
- **架构检查可选**：无法获取架构上下文时跳过 S6，不影响其他维度检测
- **兼容两种流程**：同时支持铁律流程（requirements.md）和 SpecKit 流程（spec.md）
- **与 speckit.analyze 不重复**：本命令不检查覆盖率和跨文档一致性（analyze 的职责），只检查单文档内的 SMA 精确度
