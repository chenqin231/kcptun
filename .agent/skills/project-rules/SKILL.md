# /project-rules — 添加项目自定义规范到 .agent/user.md

## 用法

- **Claude Code**：`/project-rules <规范要求>` — 将项目规范整合写入 `.agent/user.md`
- **Codex**：`$project-rules <规范要求>` 提及本 skill，或在 `/skills` 菜单中选择；也可由模型按描述自动触发
- **不要用 Claude 原生 `#`**：`#` 写入 memory/CLAUDE.md，不会进入 `.agent/user.md`，下游 `ai-rules` 分发也带不走

参数（规范要求）：Claude Code 经 `$ARGUMENTS` 传入；Codex 经提及时附带的文本或当前对话上下文传入。下文统一以 `$ARGUMENTS` 指代该规范要求。

---

## Guard

If `$ARGUMENTS` is empty, output:

```
请描述要添加的项目规范，例如：
  /project-rules 所有 API 接口必须返回统一的 JSON 格式，包含 code/message/data 三个字段
  /project-rules 禁止在业务代码中直接使用 console.log，必须使用项目统一的 logger
  /project-rules 数据库查询必须使用 ORM，禁止手写 SQL
```

Then stop. Do not proceed.

## 执行流程

**[BLOCKING CONSTRAINT]** 以下每个步骤都必须实际执行，禁止跳过。

### 步骤 1：读取现有规范

读取以下文件（能并行则并行）：

1. `.agent/user.md` — 当前项目自定义规范（L4）
2. `.agent/instructions.md`（即 `CLAUDE.md` / `AGENTS.md` 指向的核心指令）— 扫描硬约束与通用工作原则部分

如果 `.agent/user.md` 不存在，使用以下模板初始化：

```markdown
# 项目自定义规范 (L4)

> **用途**: 项目特定的自定义规范，优先级最高，可覆盖 L0-L3

---
```

### 步骤 2：冲突检测

将 `$ARGUMENTS` 中用户提出的规范要求，与核心指令（`.agent/instructions.md`）的硬约束和通用工作原则进行对比。

检查项：
- 是否与硬约束（简体中文输出 / 第一性原理）直接矛盾？
- 是否与安全编码原则冲突？
- 是否与 TDD/方案先行等强制流程冲突？
- 是否与现有 user.md 中的规范重复或矛盾？

**如果发现冲突**，输出警告并停止：

```
⚠️ 冲突检测

新规范与现有规范存在冲突：
- 冲突点：<具体描述>
- 涉及规范：<L0/L1 中的哪条规则>
- 建议：<如何调整以消除冲突>

请修改后重新执行 /project-rules
```

Then stop. Do not proceed.

**如果无冲突**，继续下一步。

### 步骤 3：优化表达

对用户的规范要求进行优化：

1. **结构化**：提取核心要点，组织为清晰的规范条目
2. **去重**：如果 user.md 已有类似规范，合并而非重复添加
3. **规范化**：
   - 使用统一的章节格式（`## 标题` + 条目列表）
   - 用"必须/禁止/建议"三级约束词
   - 每条规范一句话说清要求
4. **补充**：对模糊表述补充具体的检查标准（如有必要）

### 步骤 4：写入 .agent/user.md

将优化后的规范追加到 `.agent/user.md` 末尾（用各平台的文件编辑能力写入）。

格式要求：
- 在文件末尾追加 `---` 分隔线 + 新的规范章节
- 章节标题用 `##`，简明扼要

**如果是合并/更新已有章节**：直接修改对应章节内容，而非重复追加。

### 步骤 5：会话生效声明

**[CRITICAL]** 完成写入后，AI 必须：

1. 重新读取更新后的 `.agent/user.md` 完整内容，确认写入正确
2. 在本会话中立即将新规范纳入工作约束，后续所有操作必须遵守
3. 输出明确的生效声明（见下方报告格式）
