# /fix — Bug 修复 SOP

## 用法

- `/fix <问题描述>` — 标准模式（含 Phase 4 用户审批）
- `/fix --auto <问题描述>` — 自动模式（跳过 Phase 4）

参数：$ARGUMENTS

## Guard

解析 `$ARGUMENTS`：识别 `--auto` 标志，剩余为问题描述。问题描述为空 → 输出提示并终止：

```
请描述要修复的问题，例如：
  /fix ai-rules update --force 后 hooks 文件未同步
  /fix --auto 版本号显示为旧版本
```

## 术语（全文统一）

- **偏离点**：第一个事实 ≠ 预期 的代码位置（精确到 file:line）
- **观察事实**：来自实际运行 instrumentation 的输出
- **推断**：基于代码阅读得出的推理（非观察）

## 流程

```
Phase 0 解析 → 1 事实收敛 → 2 TDD RED → 3 设计
  → 4 审批 → 5 TDD GREEN → 6 审查 → 7 报告
```

会话级计数器 `FIX_ATTEMPT = 0`，跨 Phase 持久。

---

## Phase 1 事实收敛（无假设）

**[BLOCKING]** 本阶段产出必须全部来自实际运行输出。禁止：

- 用代码阅读声明变量值、执行路径、调用次序
- 用经验/惯例/类比代替观察
- 编写修复代码
- 输出包含禁用词（见附录）

工作流：复现 → 加 instrumentation → 第一个事实与预期分叉处即偏离点 → 差分实验证明因果。

### 1A 复现

执行用户提供的复现步骤，捕获原始输出（stdout/stderr/exit code/错误栈）。UI 问题用浏览器 MCP 截图 + 控制台日志；生产/异步问题拉日志原文。无法复现 → 向用户追问环境/版本，禁止推进。

```
### Phase 1A 产出
- 执行命令: <原文>
- 原始输出: <stdout/stderr 原文或文件路径+行号>
- 用户预期: <描述>
- 实际现象: <描述>
- 偏离焦点: <哪一行/字段/UI 元素不对>
```

### 1B 观察

让内部状态可见。

1. 从错误栈/字符串 grep 定位抛出点（Read/Grep 仅用于定位）
2. 在怀疑路径插入 instrumentation：脚本/后端用 `print`/`log`；前端用 `console.log` + 截图；生产环境查完整日志或 APM
3. 重新执行 1A 步骤，收集 instrumentation 输出
4. 产出事实表（每行有"证据等级"列）：

```
### Phase 1B 事实表
| # | 位置 file:line | 变量/调用 | 实际值 | 预期值 | 证据等级 |
|---|----------------|-----------|--------|--------|----------|
| 1 | ...            | ...       | ...    | ...    | [观察]   |
| 2 | ...            | ...       | ...    | ...    | [观察]   |
| 3 | ...            | ...       | ...    | ...    | [观察]   |
```

**通过条件**：`[观察]` 行 ≥3，**`[推断]` 行不计数**。`[推断]` 仅作辅助说明。

### 1C 定位

从事实表找**第一个 `实际 ≠ 预期`** 的行 = 偏离点。

- 多个偏离点：取**最早发生**的（上游会污染下游）
- 偏离点在第三方库内部：向上游追，定位"我方代码第一个错误调用点"
- 回归 bug：执行 `git bisect` 定位引入提交，commit hash 作为额外证据

```
### Phase 1C 产出
- 偏离点: <file:line>
- 偏离描述: <变量 V 取 A，应为 B>
- 上游观察链: <事实表行号序列，从输入到偏离点>
- git bisect: <commit hash + 标题>（如适用）
```

### 1D 证明

**最小复现**：构造 <20 行独立脚本/测试稳定触发偏离。无法构造 → 1C 定位错了，回 1B 补观察点。

**差分实验**（至少一项）：
- **反向**：修复该行为预期 → 问题消失 → 证明必要性
- **正向**：在干净环境注入该代码/数据 → 问题出现 → 证明充分性

```
### Phase 1D 产出
- 最小复现: <内联代码或文件路径>
- 复现输出: <实际运行输出原文>
- 差分类型: 反向 / 正向
- 差分操作: <具体改动>
- 差分结果: <行为变化原文>
```

### 根因陈述

```
根因: <file:line> 的 <变量/调用> 在 <触发条件> 下取值 <actual>，应为 <expected>。
证据:
  - 观察: 事实表第 N 行 <instrumentation 原文>
  - 复现: <最小复现路径或内联>
  - 差分: <改动 → 行为变化>
```

### Phase 1 通过条件（[BLOCKING]）

1. 1A 有原始复现输出
2. 1B 事实表 `[观察]` 行 ≥3
3. 1C 偏离点精确到 `file:line`
4. 1D 最小复现可运行 + 差分实验已执行
5. 输出无禁用词
6. **禁止脱离事实的猜想**：1C/1D 中**每个结论必须引用 1B 事实表行号**（格式：`（事实表 #2,#3）`）。无引用的判断视为猜想 → 回退 1B 补观察
7. **用户确认偏离点判断正确**

差分失败 → 回 1B 补观察点（上限 2 次，第 3 次熔断建议 `/speckit.specify`）。

### 根因决策锁（[BLOCKING]）

用户确认偏离点后，根因**锁定**。Phase 3-7 引用根因时只能引用 Phase 1 输出，禁止"我重新想了想"式修订。**仅当 Phase 5 实测反向证伪根因时**才能解锁回退 Phase 1。

## Phase 2 验证（TDD RED）

**约束**：只写测试，不写修复。

1. 按项目确定测试框架（bats/jest/pytest/go test 等）
2. 编写失败测试：精确复现根因行为；最小化；命名 `test_<条件>_<错误行为>`；断言检查具体表现
3. 实际执行测试，确认失败，记录运行输出

### Phase 2 通过条件

- 测试**确实失败**（有运行输出记录）
- 失败原因 = 根因预期表现，**不是环境问题**

测试通过（应失败但通过）→ 根因理解错，回 Phase 1。

## Phase 3 设计

### 3.0 加载设计人格（[BLOCKING]）

依据 Phase 3.1 变更清单的文件路径选主人格，Read `.agent/personas/<persona>.md`，按其「核心信念 + 禁止行为」组织 3.2 设计：

| 路径 / 性质                                         | 主人格            |
|----------------------------------------------------|------------------|
| `*.tsx/jsx/vue/svelte`、`components/`、`pages/`、`styles/` | frontend         |
| `*.go/py/java/rb/php`、`server/`、`api/`、`lib/`、`services/` | backend          |
| 跨模块接口 / 数据结构 / 依赖方向 变更                | architect        |
| 认证 / 鉴权 / 加密 / 输入校验                        | security-reviewer|
| CI / 构建 / 部署 / 容器 / Hook                       | devops           |
| 仅测试代码                                           | tester           |

混合场景允许加载 ≤2 个人格，主次分明。Phase 3.2 输出首行标注：`> 以 <persona> 视角设计`。

### 3.1 规模评估

| 等级 | 条件                              | 行为                                          |
|------|----------------------------------|----------------------------------------------|
| S    | ≤10 行 且 ≤1 文件                  | 直接设计                                      |
| M    | 11-50 行 或 2-3 文件               | 检查可提取的重复代码                           |
| L    | >50 行 或 >3 文件                  | **必须**参考 `design-patterns` skill + worktree 隔离 |
| XL   | 涉及模块接口/数据结构/跨模块依赖 改动 | **终止 /fix**，输出建议 `/speckit.specify`     |

### 3.2 修复方案

```
> 以 <persona> 视角设计

### Phase 3.2 产出
- 规模等级: S / M / L
- 变更清单:
  - <file:line>: <修改描述>
- 不修改的内容: <明确排除范围>
- 设计依据: <L 等级引用 design-patterns 模式>
- 副作用: <兼容性 / 测试影响 / 性能>
```

### 3.3 Linus 三问（[BLOCKING]，对齐 linus-three-questions skill）

```
### Linus 三问
1. 真问题？
   → 引用 Phase 1D 证据原文（不接受"修复 X bug"这种循环描述）

2. 更简单？
   → 必须列出 ≥1 个被放弃的更复杂方案
   → 必须给出当前方案 vs 该方案在「行数 / 抽象层 / 依赖项」至少一项的量化对比

3. 破坏什么？
   → 必须列出 ≥2 个受影响面
   → 即使结论"无影响"也要列出已检查项

结论: ✅ 通过 / ❌ 未通过，原因: <xxx>
```

任一问"通过"但未给出上述具体证据 → 视为未通过。**未通过处理**：#1 回退 Phase 1，#2 回退 3.2 重新设计，#3 补充影响分析。

### Phase 3 通过条件（[BLOCKING]）

- 3.0 已加载人格（首行标注）
- 3.2 方案完成
- 3.3 三问通过（含被放弃方案 + 受影响面 ≥2）
- 等级非 XL
- **禁止过度设计**：满足以下任一即视为过度设计 → 回退 3.2 重新设计：
  - 引入与根因无关的新抽象（接口 / 工厂 / 中间件 / 事件总线 / DI 容器）
  - 修改超出 Phase 1 偏离点上游链条的代码
  - 为"未来可能"留接口或扩展点（YAGNI 违反）
  - 3.3 第 2 问「更简单」未给出量化对比（行数 / 抽象层 / 依赖项至少一项）
  - 规模等级从 S/M 上升到 L 而无 Phase 1 证据支撑

## Phase 4 审批

**[BLOCKING]** 用户未确认前禁止进入 Phase 5。

向用户展示：根因（一句话 + 证据）/ 修复方案 / 规模 / 副作用风险。用户选择：

- 确认 → Phase 5
- 拒绝 → 回退到指定 Phase
- 修改 → 按反馈调整方案

`--auto` 模式跳过本阶段。

---

## Phase 5 修复（TDD GREEN）

### 5.1 实施修复

按 3.2 变更清单逐项修改。**最小变更**，不顺带"改进"无关代码。

### 5.2 运行目标测试

执行 Phase 2 测试。
- 通过 → 5.3
- 不通过 → `FIX_ATTEMPT += 1`；<3 回 5.1；==3 熔断

### 5.3 全量测试

- 本次引入回归 → `FIX_ATTEMPT += 1`，回 5.1
- 预存的已有失败 → 记录但不阻塞

### 5.4 边界测试

至少补充 1 个（空值 / 特殊字符 / 临界值 / 并发，按场景选）。

### Phase 5 通过条件

- 目标测试通过（有运行输出）
- 全量无本次引入回归
- 至少 1 个边界测试
- Phase 1B 临时 instrumentation 已移除（搜索 `[FIX-INSTRUMENT]` 标记或临时 log）

---

## Phase 6 审查

调用 `code-reviewer` agent，三级评审：

| 级别     | 示例                                  | 处理                              |
|---------|---------------------------------------|----------------------------------|
| CRITICAL| 安全漏洞 / 注入 / 硬编码凭证            | 回 Phase 5（`FIX_ATTEMPT += 1`）  |
| HIGH    | 函数 >50 行 / 嵌套 >4 层 / 缺错误处理   | 回 Phase 5                       |
| MEDIUM  | 命名 / 注释 / 风格                     | 标记不阻塞                        |

### Phase 6 通过条件

无 CRITICAL / HIGH。

## 熔断机制

`FIX_ATTEMPT == 3` 强制终止。输出三次失败原因，建议：

1. 运行 `/speckit.specify` 重梳需求边界
2. 评估是否需要重新设计相关模块
3. 架构层解决后再尝试修复

熔断后**禁止同会话对相同问题再执行 `/fix`**。

## 附录

### A. 禁用词清单

Phase 1 输出含以下任一词 → **强制回退 1B 补观察点**：

```
可能 | 应该是 | 估计 | 大概 | 疑似 | 看起来 | 推测 | 猜测
怀疑 | 八成 | 多半 | 假设 | 推断 | 倾向于认为
```

豁免：①引用用户原话；②向用户的提问句（如"请确认是否涉及环境差异？"）。

### B. 临时仪表管理

- 1B 加的 `print`/`log` 是临时改动，Phase 5 结束前必须移除
- 建议加 `// [FIX-INSTRUMENT]` 注释或用 git worktree 隔离
- 测试框架自带日志（`pytest -s`、`jest --verbose`）优先复用

| 组件 | 调用位置 |
