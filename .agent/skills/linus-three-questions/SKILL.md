# linus-three-questions — 编码前的质量门

## 何时激活

同时满足以下条件时激活：

1. 任务性质：功能开发 / 重构 / Bug 修复（非紧急 hotfix）
2. 方案已经确认（speckit.plan 通过、用户批准、或口头同意）
3. 即将开始写代码（下一步是 Edit / Write）

## 必须输出的格式

在第一行代码被写下之前，输出如下自检块：

```
### Linus 三问自检

1. 真问题？
   → [一句话：这到底解决了什么真实存在的问题？]

2. 更简单？
   → [代码行数、抽象层次、依赖项能否再减？用什么证据说明当前已经是最简？]

3. 破坏什么？
   → [向后兼容性、现有测试、公开 API、副作用 — 有哪些影响？]

结论：✅ 通过 / ❌ 未通过
```

## 判定准则（严格）

- 每一问必须给出**具体证据**，不接受「是的」「应该」「大概」之类的模糊回答
- 第 2 问必须至少列出一个被放弃的更复杂方案，解释为什么当前方案更简单
- 第 3 问必须列出至少 2 个受影响面（即使结论是"无影响"也要列出检查过的面）
- 任一问未通过 → **立刻停止编码**，回到方案阶段重新设计

## 豁免条件

- 用户明确指定了修改方法 **且** 变更 <10 行 **且** 单文件
- 纯文档 / 注释 / 配置 typo 修复
- 紧急生产 hotfix

## 上下游协作

- **上游**：`speckit.plan` / `speckit.tasks` 产出方案 → 本 skill 做质量门
- **下游**：通过后才能进入 `speckit.implement` 或直接 Edit
- **并列**：与 `tdd` skill 配合 — 先三问、再写失败测试、再实现

## 为什么用 Linus 三问

Linus Torvalds 审阅 Linux kernel 贡献时的核心问：解决的是真问题吗？实现得足够简单吗？会破坏什么？这三问能把大多数"过度设计"和"破坏性改动"拦在提交之前。
